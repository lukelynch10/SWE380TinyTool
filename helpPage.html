Cohesion and Coupling Calculation

The purpose for this tiny is to calculate levels of cohesion and coupling in a program. It perfroms this by asking a set of questions for each component in a program and allows the user to input the answers. This gives some infomration about the state of coupling and cohesion for each component. This data is then fed into two formulas LCOM4 for cohesion and Fenton and Melton for coupling. This is done for each component. These results for each component are then added up and averaged to determine the overall levels of cohesion and coupling in the program.

Coupling and Cohesion Coupling

Coupling is a qualitative measure of the degree to which classes are connected to one another. As classes become more interdependent, coupling increases. It is best to keep coupling as low as possible since it can lead to a lot of confusion when trying to understand and debug a program. There are different types of coupling. Here are a few examples

  • Content Coupling - Occurs when one component surreptitiously modifies data that is internal to another component.
  • Control coupling- Occurs when an operation A() invokes another operation B() and passes a control flag to B. The control flag then directs logical flow within B. This can cause an unrelated change in B that can result in the necessity to change the meaning of the control flag that A passes. Overlooking this will result in an error.
  • External coupling- Occurs when a component communicates or collaborates with infrastructure components. It is necessary but should be limited to a small number of components or classes within a system.
Cohesion

Cohesion is the qualitative indication of the degree to which a module focuses on just one thing. Cohesion implies that a component or class encapsulates only attributes and operations that are closely related to each other and to the class or component itself. There are several different types of cohesion, each having a different level of cohesion. Here are some examples list in order of the level of cohesion.

  • Functional - Exhibited primarily by operations, this level of cohesion occurs when a module performs one and only one computation and then returns a result.
  • Layer - Exhibited by packages, components, and classes, this type of cohesion occurs when a higher layer accesses the services of a lower layer, but lower layers do not access higher layers.
  • Communicational - All operations that access the same data are defined within one class. In general, such classes focus solely on the data in question, accessing and storing it.
LCOM4 Formula

The LCOM4 formula is a metric used to measure the levels of cohesion in a class. It does this by measuring the number of connected components in a class. A connection is a set of related methods. Ideally there should only be one connected component in each class. The formula is below. In order to use the formula you must:

  • First calculate the mean number of access per attribute across all methods in the class.
  • From there you need to count the number of distinct attributes accessed for each method in the class (Ni)
  • Then you need to calculate the sum of the squares of these counts.
  • After you need to calculate the mean number of method invocations across all methods in the class
  • Then you need to calculate the sum of the squares of these counts.
  • Now you need to subtract the sum of the squares of method invocations from the sun of the squares of attributes accessed.
  • If the result is negative set it to 0
  • Then divide the result by the product of the mean number of access per attribute (M) and the mean number of method invocations
  • The final result is LCOM4
  • LCOM4 = 1 - Indicates a good level of cohesion
  • LCOM4 >= 2 - Indicates the class should be split into smaller classes
  • LCOM4 = 0 - Indicates there are no methods in the class which is bad.
Fenton and Melton Formula

The Fenton and Melton metric is used to measure the degree of an architectures coupling. It does this by obtaining the number of direct dependencies that exist between two components as well as a 1-5 value that indicates the level of coupling that exists between the two components. These values are plugged into the formula below with:

C( a , b ) = i + n / (n + 1)
  • i - representing the level of coupling
  • n - representing the number of direct dependencies.
  • a & b - representing the two components
  • C - representing the level of coupling

The lower the value the lower amount of coupling there is. Lower values are good since they indicate a lower amount of coupling which is what you want your program to have.
